# Conventions

## Общие правила

- Реализация только через активный execution plan.
- Никаких догадок о Pumax API без зафиксированного контракта.
- Минимальный дифф на задачу.
- Любое изменение публичного поведения требует обновления docs.

## Именование

- `telegram_user_id`: идентификатор пользователя Telegram.
- `binding`: привязка Telegram-пользователя к MAX-сессии.
- `max_chat_id`: идентификатор личного чата в MAX.
- `telegram_topic_id`: идентификатор topic внутри личного диалога с ботом.
- `reconcile`: полная сверка состояния MAX и Telegram для одного пользователя.
- `backfill`: подтягивание последних `N` сообщений в новый или восстановленный topic.

## Стиль задач

Каждый execution plan обязан содержать:
- контекст;
- конкретные артефакты для изменения;
- риски;
- верификацию;
- явное указание, какие документы должны быть обновлены вместе с кодом.

## Стиль будущих тестов

- Тесты зеркалят структуру `src/`.
- Изоляция пользователя тестируется отдельно и явно.
- Сценарии `/start` и `reauth_required` покрываются как happy path, так и edge cases.
- Для неподдерживаемых вложений нужен тест на fallback-описание, а не на молчаливую потерю.
