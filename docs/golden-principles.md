# Golden Principles

## Архитектурные аксиомы

1. Входное событие без allowlist не должно запускать никакую бизнес-логику.
2. Один Telegram-пользователь может быть связан только с одним MAX-аккаунтом.
3. Любое действие выполняется строго в контексте конкретного `telegram_user_id`.
4. `/start` это не только вход, но и reconcile.
5. Истекшая MAX-сессия не переавторизуется автоматически.
6. Потеря сообщения без явного уведомления недопустима.
7. Любой новый topic создается вместе с backfill последних `N` сообщений.
8. Telegram main chat служит только для команд и сервисных ответов.
9. Неподдерживаемый контент конвертируется в человекочитаемое служебное сообщение.
10. Архитектурные границы важнее скорости первичной поставки.

## Правила первой версии

- Поддерживаются только личные MAX-чаты.
- Группы, каналы, удаления и редактирования отложены.
- SQLite достаточно до появления явной нагрузки или требований миграции.
- Docker и конфиг окружения обязательны с первого дня.
