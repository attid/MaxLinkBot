# MLB-002: Domain Model And SQLite Schema

## Контекст

Система критически зависит от корректной модели binding, topic mapping и журналов доставки. До интеграции с Telegram и MAX нужно зафиксировать доменные сущности и схему хранения.

## Результат задачи

- доменные модели пользователя, binding, MAX chat, Telegram topic, message link, sync cursor, audit event;
- схема SQLite и миграции;
- репозитории или порты доступа к данным;
- тесты на инварианты изоляции и допустимые переходы состояний binding.

## Шаги

1. Описать таблицы и ключи SQLite.
2. Зафиксировать состояния binding: `active`, `reauth_required`, `disabled`.
3. Ввести уникальные ограничения:
   один binding на `telegram_user_id`, один mapping на пару `telegram_user_id + max_chat_id`.
4. Описать курсоры и аудит событий.
5. Написать тесты на недопустимость пересечения пользовательских маппингов.

## Риски и открытые вопросы

- Нужна явная стратегия idempotency для повторной доставки сообщений.
- Нужно заранее определить, где хранить внешние идентификаторы сообщений MAX и Telegram.

## Верификация

- схема SQLite покрывает все обязательные сущности из ТЗ;
- тесты доказывают запрет на смешивание данных разных пользователей;
- переходы `active -> reauth_required` и обратный путь документированы и проверены.
