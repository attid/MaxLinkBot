# MLB-012: Human-Readable Chat Titles And Message Labels

## Контекст

После стабилизации `/start` и `reconcile` topics начали создаваться, но UX остается слабым:
- для части чатов topic создается с fallback title вида `Chat 218233000`, что не помогает понять, кто это;
- входящие сообщения из MAX приходят без явной маркировки отправителя и времени, поэтому в topic нельзя быстро понять, кто именно написал сообщение.
- probe-проверка удалённых private topics через отправку тестового сообщения оказалась ненадёжной: Telegram не всегда возвращает различимую ошибку.

Перед изменением кода нужно проверить, какие поля реально доступны из `pymax` для чатов и сообщений, и выбрать стабильный формат отображения.

## План изменений

1. [ ] Через `local_max_test.py` и/или локальную диагностику проверить, какие поля `pymax` возвращает для chat title, peer/user name и метаданных сообщения.
2. [ ] Зафиксировать минимальный формат отображаемого названия topic:
   приоритетно использовать человеческое имя/название чата, fallback оставить только последним вариантом.
3. [ ] Зафиксировать формат входящего сообщения для Telegram topics:
   `[Имя DD.MM.YY HH:MM] текст`
4. [ ] Добавить/обновить unit-тесты для title resolution и inbound message rendering.
5. [ ] Реализовать минимальные изменения в `src/infrastructure/max/adapter.py`, `src/application/reconcile/service.py` и/или `src/application/routing/inbound.py`.
6. [ ] Изменить reconcile semantics:
   `/start` только создает отсутствующие mappings, а отдельная команда `/resync` пересоздает все topics заново.
7. [ ] Добавить меню-команду `/resync` и handler для полного rebuild topics.
8. [ ] Обновить docs, если после проверки появится важный новый контракт `maxapi-python`.
9. [ ] Прогнать целевые тесты и ручную локальную проверку через Docker или `local_max_test.py`.

## Риски и открытые вопросы

- `pymax` может не отдавать удобное поле имени чата в `fetch_chats()`, и тогда понадобится дополнительная нормализация или lookup по сообщениям.
- Нужно аккуратно форматировать служебные и media-сообщения, чтобы маркировка не ломала текущий fallback.
- Важно не сломать существующий backfill и cursor update при изменении структуры message dict.
- Полный rebuild topics через `/resync` может создавать новые topics поверх старых живых, если пользователь не удалил их вручную; это допустимый компромисс ради предсказуемого восстановления.

## Верификация

- topics получают человекочитаемые названия, если MAX отдает хоть какие-то name/title поля;
- если человеческое имя недоступно, fallback остается стабильным и предсказуемым;
- новые входящие и backfill-сообщения в Telegram topic начинаются с префикса вида `[Имя 22.03.26 14:35]`;
- `/start` не пересоздает уже существующие mappings;
- `/resync` создает новый набор topics и обновляет mappings для всех текущих MAX chats;
- целевые pytest-тесты проходят;
- ручная проверка в Docker подтверждает читаемые titles и маркировку сообщений.
