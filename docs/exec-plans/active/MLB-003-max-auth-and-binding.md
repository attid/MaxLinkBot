# MLB-003: MAX Authorization And Binding Lifecycle

## Контекст

Без устойчивого сценария авторизации система не сможет безопасно обслуживать нескольких пользователей. Нужно изолированно провести каждого разрешенного Telegram-пользователя через привязку его MAX-аккаунта.

## Результат задачи

- application-сценарий авторизации через MAX API c инфраструктурной оберткой поверх `maxapi-python`;
- сохранение успешного binding в SQLite;
- восстановление существующего binding;
- уведомление и остановка при истекшей сессии.

## Шаги

1. Описать интерфейс авторизации к MAX API и место использования `maxapi-python` в инфраструктурном слое.
2. Реализовать сценарий создания binding для нового пользователя.
3. Реализовать повторную авторизацию для `reauth_required`.
4. Ограничить процесс только allowlist-пользователями.
5. Добавить тесты на отказ неразрешенному пользователю и на повторную привязку существующего пользователя.

## Риски и открытые вопросы

- Реальный UX авторизации зависит от того, что именно умеет MAX API.
- Важно не допустить частично сохраненных binding при падении в середине flow.

## Верификация

- новый разрешенный пользователь может получить binding;
- неразрешенный пользователь не доходит до сценария авторизации;
- истекшая сессия переводится в `reauth_required` и требует явного нового `/start`.
