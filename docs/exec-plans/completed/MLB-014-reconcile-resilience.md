# MLB-014: Reconcile Resilience On Topic/Backfill Failures

## Контекст

Во время `/resync` единичная ошибка доставки в Telegram topic сейчас валит весь reconcile flow.
Практически это уже воспроизвелось на `TelegramBadRequest: message thread not found` во время backfill.

Такое поведение плохое по двум причинам:
- один битый/удалённый/нестабильный topic не должен останавливать rebuild остальных чатов;
- единичный сбой отправки одного сообщения не должен ронять backfill целиком.

Нужна fail-safe логика:
- логировать каждую ошибку с достаточным контекстом;
- продолжать обработку остальных сообщений и чатов;
- не обновлять cursor для topic, где backfill был частично или полностью неуспешен.

## План изменений

1. [x] Добавить regression tests на backfill-message failure и per-chat reconcile failure.
2. [x] Изолировать отправку отдельных backfill сообщений: log-and-continue вместо raise.
3. [x] Возвращать из backfill признак полного/неполного успеха.
4. [x] Не обновлять cursor, если backfill завершился с ошибками.
5. [x] Изолировать reconcile per MAX chat: один проблемный чат не валит весь `/resync`.
6. [x] Прогнать целевые pytest-тесты.

## Риски и открытые вопросы

- Если topic реально удалён или невалиден, ошибки будут повторяться для нескольких сообщений подряд; это ожидаемо, но не должно останавливать остальные чаты.
- Нельзя обновлять cursor после частичного backfill, иначе сообщения могут быть потеряны без повторной попытки.
- Нужно сохранить текущий `AuthError` path как особый случай, а не глушить его бездумно.

## Верификация

- `TelegramBadRequest` на одном сообщении backfill не валит `/resync`;
- следующий message в том же backfill всё ещё обрабатывается;
- следующий чат в reconcile всё ещё обрабатывается;
- cursor не обновляется для неуспешного backfill;
- целевые pytest-тесты проходят.
