# 0001: MAX Integration Library Selection

## Контекст

Проекту нужен зафиксированный способ интеграции с экосистемой MAX, чтобы последующие исполнители не выбирали библиотеку заново при старте реализации. В пользовательском требовании явно указан репозиторий `https://github.com/love-apples/maxapi`.

Дополнительно по публичной странице репозитория видно, что:
- библиотека предназначена для разработки чат-ботов для MAX;
- это Python-библиотека;
- у проекта есть документация и пакетная установка через `pip install maxapi`.

## Решение

В первой версии проекта библиотекой интеграции с MAX фиксируется `love-apples/maxapi`.

Правило использования:
- прямое использование допускается только в `infrastructure/max/`;
- `domain/` и `application/` не зависят от `maxapi` напрямую;
- внешний контракт библиотеки должен быть закрыт собственным адаптером, чтобы при необходимости заменить реализацию без переписывания бизнес-логики.

## Последствия

- старт реализации ускоряется: выбор библиотеки больше не обсуждается;
- архитектура обязана держать `maxapi` за портом/адаптером, а не размазывать по проекту;
- в `MLB-001` и `MLB-003` нужно явно проверить, какие части авторизации и обмена сообщениями закрываются библиотекой, а какие останутся на стороне MAX API или собственной обвязки;
- если later выяснится, что `maxapi` не покрывает нужные сценарии, потребуется новый ADR, а не тихая замена.
