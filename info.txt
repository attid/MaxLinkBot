Ниже цельное ТЗ на первую версию бота-шлюза, только по тому, что мы уже зафиксировали в этой сессии.

## Общее назначение

Нужно разработать Telegram-бота на Python, работающего в Docker и использующего aiogram, который служит персональным шлюзом между Telegram и Messenger MAX через Pumax.

Бот не общий для одного MAX-аккаунта, а мультипользовательский по модели:

* один Telegram-бот;
* несколько разрешённых Telegram-пользователей;
* у каждого разрешённого Telegram-пользователя свой собственный MAX-аккаунт;
* связь строгая: **1 Telegram user = 1 MAX account**.

Бот должен вести несколько независимых MAX-сессий одновременно, по одной на каждого разрешённого Telegram-пользователя, без смешивания данных между ними.

---

## Основные технологические требования

Бот должен быть реализован:

* на **Python**;
* с использованием **aiogram**;
* в **Docker**;
* с локальной базой данных **SQLite**.

---

## Модель доступа

Список разрешённых Telegram-пользователей задаётся через параметры контейнера, например через env.

Бот должен принимать сообщения только от Telegram user_id, входящих в allowlist.

Все остальные пользователи должны жёстко игнорироваться без какой-либо бизнес-логики.

Allowlist определяет только право пользоваться ботом в Telegram.
После этого каждый пользователь отдельно привязывает **свой** MAX-аккаунт.

---

## Модель привязки аккаунтов

Для каждого разрешённого Telegram-пользователя должна существовать отдельная запись привязки к MAX-аккаунту.

Поддерживается только один MAX-аккаунт на одного Telegram-пользователя.

Несколько MAX-аккаунтов на одного Telegram-пользователя в первой версии не поддерживаются.

Привязка должна храниться в локальной базе и содержать не просто факт существования пользователя, а именно рабочее состояние его MAX-сессии.

---

## Поведение команды `/start`

Команда `/start` является основной пользовательской точкой входа.

Логика должна быть такой:

### Если Telegram user не входит в allowlist

Бот его игнорирует.

### Если Telegram user входит в allowlist, но привязки MAX ещё нет

Бот запускает сценарий авторизации этого пользователя в его собственный MAX-аккаунт через Pumax.

### Если привязка MAX уже есть и валидна

Бот не просто отвечает “ты уже авторизован”, а выполняет пользовательский refresh/reconcile:

* проверяет авторизацию;
* получает список личных чатов MAX этого пользователя;
* проверяет соответствующие topics в Telegram;
* создаёт недостающие topics;
* восстанавливает потерянные или удалённые topics;
* подтягивает хвост истории в новые или заново созданные topics;
* после этого продолжает нормальную работу.

То есть во внешнем интерфейсе `/start` выполняет роль “войти или обновить”.

---

## Отдельная команда `/refresh`

Как отдельная пользовательская команда в первой версии не обязательна.

Но внутри кода должна существовать отдельная функция или сервисный метод refresh/reconcile, который использует `/start` и который можно будет потом при необходимости вынести в отдельную команду.

---

## Авторизация MAX

Если у пользователя нет действующей привязки, бот должен запускать сценарий авторизации MAX через Pumax.

Сам механизм авторизации считается внешним по отношению к боту: бот должен уметь провести пользователя по этому сценарию и затем сохранить успешную привязку/сессию.

---

## Истечение MAX-сессии

Если при фоновой проверке или при очередной попытке обращения к MAX выясняется, что сессия протухла, бот должен:

* перевести привязку в состояние вроде `reauth_required`;
* один раз отправить пользователю служебное сообщение о том, что авторизация MAX истекла;
* в этом сообщении дать кликабельный `/start` или кнопку наподобие «Начать авторизацию».

После этого бот не должен бесконечно продолжать опрашивать этот MAX-аккаунт.

До тех пор, пока пользователь сам не нажмёт `/start` или кнопку и не пройдёт повторную авторизацию, бот должен считать этот аккаунт остановленным и не выполнять по нему рабочий опрос.

---

## Пространство пользователя в Telegram

Для каждого Telegram-пользователя бот работает в его **личном диалоге** с ботом.

Внутри этого личного диалога бот создаёт **topics**, и эти topics соответствуют чатам именно его MAX-аккаунта.

Topics одного пользователя не должны пересекаться с topics другого пользователя.

То есть каждый пользователь видит только свои собственные MAX-чаты, представленные как topics внутри его личного чата с ботом.

---

## Типы чатов MAX в первой версии

В первой версии поддерживаются только **личные чаты MAX (1:1)**.

Группы, каналы и иные типы диалогов MAX пока не поддерживаются.

---

## Отображение MAX-чатов в Telegram

Каждый личный чат MAX должен соответствовать одному topic в Telegram.

Связь должна быть стабильной и храниться в базе данных.

Если у пользователя в MAX появился новый личный чат, бот должен автоматически:

* создать новый topic в его Telegram-диалоге с ботом;
* сохранить маппинг;
* подтянуть последние N сообщений;
* затем продолжить доставку новых сообщений в этот topic.

---

## Поведение при удалении topic

Если topic был удалён вручную, при следующем `/start` бот должен:

* заново сверить список чатов MAX;
* понять, что topic отсутствует;
* создать его снова;
* снова привязать к соответствующему MAX-чату;
* подтянуть хвост истории.

Это нужно как для реального восстановления, так и для удобной отладки.

---

## Подтягивание истории

При первом создании topic или при его повторном создании бот должен подтягивать не только новые сообщения с текущего момента, но и последние N сообщений из соответствующего MAX-чата.

Стартовое значение N — **5**.

Эта величина должна быть вынесена в конфиг как константа.

Идея нужна и для восстановления после удаления topic, и для тестирования передачи текста, картинок и других вложений.

В первой версии достаточно логики “подтянуть последние N сообщений”, без сложного ограничения по возрасту.

---

## Поведение основного диалога с ботом

Основной диалог с ботом предназначен только для:

* команд;
* служебных сообщений;
* уведомлений об истечении авторизации;
* результатов `/start` и системных действий.

Если пользователь пишет обычный текст **не внутри topic**, бот не должен отправлять этот текст в MAX.

Вместо этого бот должен отправить служебное сообщение наподобие:

“Пишите внутри нужного топика. Для обновления списка чатов используйте /start”.

---

## Двусторонняя маршрутизация сообщений

### MAX → Telegram

Если в личном MAX-чате пользователя появляется новое сообщение, бот должен:

* определить, какому MAX-чату оно принадлежит;
* найти или создать соответствующий Telegram topic этого же пользователя;
* отправить сообщение в этот topic;
* сохранить в БД запись о доставке и связи сообщений.

### Telegram → MAX

Если пользователь пишет сообщение внутри topic, бот должен:

* определить, к какому MAX-чату привязан этот topic;
* отправить сообщение в соответствующий чат его MAX-аккаунта;
* сохранить запись о доставке в БД.

---

## Правило изоляции

Пользователь Telegram A не должен иметь возможности:

* видеть чаты MAX пользователя B;
* писать в чаты MAX пользователя B;
* случайно использовать чужой topic или чужой маппинг.

Вся маршрутизация, фоновые проверки и reconcile должны выполняться строго в контексте конкретного Telegram-пользователя и его собственного MAX-аккаунта.

---

## Поддержка контента

Удаления и редактирования в первой версии не синхронизируются.

Если что-то удалили в Telegram — бот это в MAX не переносит.
Если что-то удалили в MAX — бот это в Telegram не переносит.

Основной приоритет — корректная доставка новых сообщений.

### Текст

Текстовые сообщения должны пересылаться полностью.

### Медиа и вложения

Аудио, видео, изображения и прочие вложения нужно пробовать передавать по мере возможностей реализации через Pumax и Telegram.

Если конкретный тип вложения переслать нельзя, бот не должен молча терять сообщение. Он должен отправить в topic или в соответствующий поток служебное описание, например:

* “Пользователь MAX 5508 отправил стикер”
* “Получено неподдерживаемое вложение”
* “Не удалось переслать файл такого-то типа”

То есть потеря данных без уведомления недопустима.

---

## Хранение данных

Для первой версии достаточно SQLite.

База должна хранить как минимум:

* список разрешённых/известных Telegram-пользователей;
* привязки `telegram_user_id -> MAX account/session`;
* известные личные чаты MAX;
* соответствия `MAX chat -> Telegram topic`;
* журнал обработанных и отправленных сообщений;
* состояния синхронизации и авторизации;
* аудит ошибок и служебных событий.

---

## Состояния привязки

У привязки MAX-аккаунта должно быть явное состояние, например:

* `active`
* `reauth_required`
* `disabled`

Для первой версии особенно важны `active` и `reauth_required`.

---

## Фоновая работа

Бот должен выполнять фоновую проверку новых сообщений и новых чатов MAX для каждого активного пользователя.

Если появляется новый личный чат MAX, бот автоматически создаёт для него topic и подтягивает хвост истории.

Если авторизация истекла, бот перестаёт дальше активно проверять этот аккаунт до ручного запуска повторной авторизации пользователем.

---

## Docker и конфигурация

Бот должен запускаться в Docker-контейнере.

Через конфигурацию контейнера должны задаваться как минимум:

* Telegram bot token;
* список разрешённых Telegram user_id;
* путь к SQLite;
* параметры Pumax;
* интервал фоновой проверки;
* значение константы для backfill истории, стартово `5`.

---

## Что обязательно входит в первую версию

Первая версия должна включать:

* Python;
* aiogram;
* Docker;
* SQLite;
* allowlist по Telegram user_id;
* `/start` как вход + refresh;
* персональную привязку одного MAX-аккаунта на одного Telegram-пользователя;
* работу только с личными чатами MAX;
* topics в личном чате пользователя с ботом;
* автоматическое создание недостающих topics;
* подтягивание последних 5 сообщений при создании/восстановлении topic;
* двустороннюю передачу новых сообщений;
* уведомление о протухшей авторизации;
* отказ от автоматической переавторизации без действия пользователя;
* служебный ответ на текст, присланный вне topic;
* хранение маппингов и состояний в базе.

---

## Что в первую версию не входит

В первой версии не требуется поддерживать:

* несколько MAX-аккаунтов на одного Telegram-пользователя;
* группы и каналы MAX;
* синхронизацию удалений;
* синхронизацию редактирований;
* сложную систему ролей поверх allowlist;
* полноценную внешнюю команду `/refresh`, если `/start` уже закрывает этот сценарий;
* гарантированную поддержку всех типов вложений без fallback-описания.

---

## Ключевая модель системы в одной строке

Нужно сделать **персональный многопользовательский Telegram-шлюз**, где один бот обслуживает нескольких разрешённых Telegram-пользователей, и каждый из них внутри своей лички с ботом получает topics, отражающие его собственные личные чаты в MAX, с отдельной авторизацией, отдельной сессией и полной изоляцией от других пользователей.
